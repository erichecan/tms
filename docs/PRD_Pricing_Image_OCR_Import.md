# 报价 Excel 图片报价 OCR 导入产品需求文档（PRD）

**文档版本**：v1.0  
**创建日期**：2026-03-13  
**状态**：草稿（一次性导入工具需求，用于数据初始化）  

---

## 一、背景与目标

### 1.1 背景

- 现有报价数据主要来源于根目录下两个 Excel：
  - 《合作单位报价卡.xlsx》
  - 《2026年转运汇总表.xlsx》
- 通过 `scripts/import_excel_to_prod.py` 已经将**大部分结构化单元格数据**导入生产库，对应表包括：
  - `pricing_matrices` 等报价表；
  - `containers`、`container_items` 等转运相关表。
- 但在《合作单位报价卡.xlsx》中，**部分 sheet 的报价不是以单元格形式存在，而是以图片（截图/扫描件）形式嵌入**，现有导入逻辑完全忽略了这些图片内容，导致：
  - 这些图片报价**没有进入数据库**；
  - 存在报价不完整、后续快速报价/运价分析不准确的风险。

### 1.2 目标

本 PRD 仅针对一次性数据导入能力的增强，目标是：

- **补齐图片报价数据**：自动识别并导入约 20 张图片中的报价信息，落地到现有 `pricing_matrices`（及相关表）中。
- **全自动执行**：在运行导入脚本时，图片报价无需人工干预即可完成识别与入库。
- **自建 OCR**：严格使用本地自建 OCR 引擎（例如 Tesseract），不依赖任何云端 OCR 服务。
- **不影响线上运行逻辑**：仅增强 `scripts/import_excel_to_prod.py` 这类离线导入工具，不改动线上 API 行为。

---

## 二、范围与约束

### 2.1 范围

**在《合作单位报价卡.xlsx》维度：**

- 对象：所有 sheet 中**以图片形式存储的报价表格**。
- 内容：FC 仓号（例如 `YYZ3/YYZ4/YHM1` 等）、板数档（1–4 板、5–14 板、15–28 板、散板等）、对应金额（全包价或每板单价）。
- 输出：转换为与现有文本报价同样结构的记录，写入：
  - `pricing_matrices`：核心目标表。
  - 如有必要，可在后续迭代中扩展至 `container_allins` 等其它报价表。

**在代码与工具维度：**

- 仅涉及：
  - `scripts/import_excel_to_prod.py`（导入脚本增强）；
  - 新增必要的解析与 OCR 辅助函数；
  - 可选的测试脚本 `scripts/test_import_excel_to_prod.py`（用于本地验证）。

### 2.2 不在范围内

- 不改动线上接口（`/api/pricing/*`）、前端页面或运行时报价计算逻辑。
- 不实现“导入过程中人工审核 UI”；本次聚焦于一次性全自动导入。
- 不修改 Excel 源文件本身（不在 Excel 内写入公式或额外 sheet）。

### 2.3 业务与技术约束

- **自建 OCR**：必须使用本地可控的 OCR 引擎（例如系统安装的 `tesseract-ocr`），禁止依赖云厂商 OCR API。
- **图片数量与格式**：
  - 约 20 张图片；
  - 可能存在不同的排版样式、分辨率和清晰度；
  - 图片中通常包含中英文混排（中文说明 + 英文仓号代码）。
- **时间维度**：该需求是一次性数据补齐，完成导入后日常报价维护将完全在 TMS 系统内完成。

## 四、业务需求描述

### 4.1 当前行为（As-Is）

- 导入脚本 `scripts/import_excel_to_prod.py`：
  - 对于《2026年转运汇总表.xlsx》：
    - 使用 `openpyxl` 按 sheet、行逐行解析，生成 `containers`、`container_items` 记录并入库。
  - 对于《合作单位报价卡.xlsx》：
    - 函数 `parse_quote_sheet(ws)` 仅基于单元格文本解析：
      - 通过“仓号”列识别 FC 代码；
      - 通过包含“1-4”“5-14”“15-28”等字样的表头列识别板数档；
      - 按行解析金额，生成 `destination_code` + `pallet_tier` + `base_price` / `per_pallet_price`；
    - `import_quote` 对每个 sheet 调用 `parse_quote_sheet(ws)`，若返回为空则直接跳过该 sheet。
- 对于嵌入图片的 sheet：
  - `openpyxl` 的 `iter_rows(values_only=True)` 无法获取图片内容；
  - `parse_quote_sheet(ws)` 返回空；
  - 导致该 sheet 中的报价**全部未入库**。

### 4.2 期望行为（To-Be）

在不改变现有“单元格解析”逻辑的前提下，导入脚本需要新增以下能力：

1. **自动检测并提取 Excel 中的嵌入图片**：
   - 针对《合作单位报价卡.xlsx》的每个 sheet：
     - 在现有文本解析失败（结果为空）时，尝试查找并提取该 sheet 上所有嵌入图片的二进制内容。
2. **对图片执行 OCR，获取可解析的文本**：
   - 使用本地安装的 Tesseract OCR，通过 Python 封装库（如 `pytesseract`）执行识别；
   - 默认语言配置为简体中文 + 英文（如 `chi_sim+eng`），可通过环境变量覆盖。
3. **将 OCR 文本解析为报价行数据**：
   - 解析目标与现有 `parse_quote_sheet` 保持一致：
     - 提取目的仓代码（如 `YYZ3`, `YOO1`, `YHM1` 等）；
     - 按板数档（1–4、5–14、15–28、散板等）解析对应金额；
     - 生成字段：`destination_code`, `pallet_tier`, `base_price`, `per_pallet_price`。
4. **统一入库流程**：
   - `import_quote` 按以下顺序处理：
     1. 调用 `parse_quote_sheet(ws)`（文本模式）；
     2. 若结果为空，再调用 `parse_quote_sheet_from_images(ws)`（图片 + OCR 模式）；
     3. 将二者任一返回的报价行列表统一写入 `pricing_matrices`，保持与现有逻辑一致。
5. **异常与容错**：
   - 单个 sheet 的 OCR 或解析失败**不会中止整个导入过程**，仅打印告警日志；
   - 对明显有金额但无法入库的行进行日志记录，便于后续人工核对；
   - 如果 OCR 结果完全无法解析为结构化报价，也不写入数据库，但需在日志中明确标注该 sheet 被跳过。

## 五、功能需求明细

### 5.1 Excel 图片提取

**需求 ID：OCR-IMG-1**

- 在导入过程中，对《合作单位报价卡.xlsx》中的每个 sheet：
  - 若 `parse_quote_sheet(ws)` 返回空列表：
    - 使用 `openpyxl` 提供的接口（例如 `ws._images`）获取嵌入的图片对象；
    - 尝试将每个图片对象转换为 `bytes`，用于后续 OCR；
    - 若某 sheet 不包含任何图片，则按当前逻辑跳过。

**验收标准：**

- 对至少一个已知“只有图片没有文本报价”的 sheet，能够成功提取到 ≥1 张图片的数据（通过日志确认）。

### 5.2 OCR 文本识别

**需求 ID：OCR-TEXT-1**

- 使用系统安装的 Tesseract OCR，通过 Python 封装执行图片识别：
  - 默认语言配置 `chi_sim+eng`（可通过环境变量覆盖）；
  - 对每张图片调用一次 OCR，返回一个多行文本字符串。
- 若单张图片 OCR 出错（例如 Tesseract 报错）：
  - 记录 warning 日志；
  - 跳过该图片继续处理其它图片。

**验收标准：**

- 在测试环境中，至少对一张图片获取到了非空文本输出；
- 当 Tesseract 未安装或配置错误时，脚本给出明确错误提示，而不是静默失败。

### 5.3 OCR 文本解析为报价数据

**需求 ID：OCR-PARSE-1**

- 实现一个纯文本解析规则，用于把 OCR 输出转换为报价行，要求：
  - 能够识别常见仓号代码（例如 `YYZ3`, `YYZ4`, `YHM1`, `YOO1` 等），支持正则匹配；
  - 能够从同一行中识别多组金额值，按照既定顺序映射到板数档（如 `1-4板`、`5-14板`、`9-13板`、`15-28板`、`散板`）；
  - 对无法成功转换为数字的 token 安全跳过；
  - 返回的结构需与 `parse_quote_sheet` 返回结构对齐：
    - `{"destination_code": ..., "pallet_tier": ..., "base_price": float 或 None, "per_pallet_price": float 或 None}`。

**验收标准：**

- 对 2–3 个经典 OCR 文本样例，解析结果中的：
  - `destination_code` 与原图片中的仓号代码一致；
  - 不同板数档的金额映射到正确的 `pallet_tier` 上；
  - 解析过程中未抛出未捕获异常。

### 5.4 与现有导入逻辑的集成

**需求 ID：OCR-INTEG-1**

- 在 `import_quote` 中集成 OCR 模式：
  - 先执行现有文本模式 `parse_quote_sheet(ws)`；
  - 若结果为空，再尝试 `parse_quote_sheet_from_images(ws)`；
  - 将任一模式得到的结果统一入库：
    - 继续使用 `pricing_matrices` 表；
    - `customer_id` 的生成规则与当前脚本中按 sheet 名生成的逻辑保持一致；
    - `vehicle_type` 默认仍使用 `"STRAIGHT_26"`（与现有逻辑一致）。

**验收标准：**

- 对原本只有图片报价的 sheet，在导入完成后，`pricing_matrices` 中能查询到对应 `customer_id + destination_code + pallet_tier` 的记录；
- 对原本已有单元格报价的 sheet，导入行为不受影响（不产生重复或冲突）。

### 5.5 日志与可追溯性

**需求 ID：OCR-LOG-1**

- 在导入过程中增加必要日志，至少包括：
  - 每个处理的 sheet 名称；
  - 文本模式与图片模式分别解析出的记录数量；
  - 对于解析失败或疑似异常的情况，包含：
    - sheet 名；
    - 简化后的 OCR 文本样例（可以截断）；
    - 解析失败的原因简述。

**验收标准：**

- 在一次完整导入执行日志中，管理员能清晰看到：
  - 哪些 sheet 使用了 OCR 模式；
  - 各自导入了多少条报价记录；
  - 是否存在完全无法解析的 sheet（若有，日志中有明确记录）。

## 六、非功能需求

### 6.1 性能

- 报价图片总数约为 20 张，单次导入执行时间可以在数十秒级内完成，即可接受；
- OCR 调用为 I/O 密集且 CPU 消耗较高的操作，但由于为一次性离线脚本，不对线上服务性能产生影响；
- 无需引入复杂的并发或队列机制，按图片顺序串行识别即可。

### 6.2 稳定性与容错

- Tesseract 未安装或语言包缺失时，脚本需在启动阶段给出**明确错误提示**，引导运维补装依赖；
- 单张图片或单个 sheet 的识别/解析失败不得中断其它 sheet 的处理；
- 所有外部调用（OCR）异常必须有 try/except 包裹。

### 6.3 安全与数据一致性

- 本次修改只影响离线导入脚本，不涉及对线上接口的权限控制变更；
- 导入逻辑仍采用当前的 upsert + 唯一约束策略，确保不会产生重复记录；
- 若 OCR 导入导致某些仓号/板数档价格与人工预期不同，由后续业务对账决策是否进行手动调整，本脚本不承担业务纠偏。

## 七、实施建议与后续工作

### 7.1 实施建议

- 在开发与测试阶段，优先针对**测试库**执行导入脚本，避免直接操作生产数据；
- 在 PR 中对 `scripts/import_excel_to_prod.py` 的改动进行详细代码评审，重点关注：
  - OCR 错误处理是否完备；
  - 文本解析规则是否足够健壮；
  - 不会影响现有成功导入的文本报价。

### 7.2 后续可能的迭代（超出本次范围，仅作记录）

- 增加简单的“导入报告”导出能力（例如输出到 CSV 或日志），列出每个 sheet 与每条报价的解析结果；
- 针对 OCR 准确率不高的场景，引入**半自动校对**界面，让业务人员在系统内对识别结果进行快速确认后再入库；
- 若未来仍有合作伙伴使用图片形式提供报价，可以考虑：
  - 直接约定统一的系统内导入模板；
  - 或搭建上传图片 → OCR → 校对 → 入库的常驻工具，而不仅仅是一次性脚本。

## 八、验收标准（汇总）

- [ ] 对原本只有图片报价的 sheet，导入后 `pricing_matrices` 中存在对应目的仓 + 板数档的记录，金额与 Excel 图片中一致（允许个别 OCR 误差，需人工抽检确认）。
- [ ] 对原本已有文本报价的 sheet，导入结果与当前版本保持一致，无额外重复记录。
- [ ] 导入脚本在 Tesseract 正确安装的环境中可以一次性跑完，无未捕获异常。
- [ ] 导入日志中清晰标注了每个 sheet 的处理模式（文本/图片）、导入条数及潜在异常。
