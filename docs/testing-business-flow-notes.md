## TMS 核心业务 & 测试视角理解（转运相关） — 2026-03-13 20:30:00

> 说明：本文件用于沉淀当前对 TMS 转运相关业务流程、Excel 数据、以及现有后端测试设计之间关系的理解，方便后续补齐功能与测试用例设计。

---

### 1. 现有测试文档与系统抽象

- **测试文档来源**：`TMS_TESTING_PLAN.md`
- **核心抽象**：
  - 客户视角：发货 / 需求流（创建运单）
  - 调度视角：运费智能计算、运单指派、Trip 管理
  - 司机 / 车辆视角：司机端操作、位置上报、数据同步
  - 财务视角：应收 / 应付结算
  - 系统视角：RBAC 权限
- **后端集成测试现状**：
  - 以 API 为主线，从 `POST /waybills` 开始串联完整闭环
  - 存在 3 组 P0 完整闭环用例（创建→指派→运输→签收→财务）

**结论（从测试文档出发）：**

- 在当前技术文档与测试视角下，系统的“业务起点”被定义为：**在系统内创建一张运单（Waybill Creation）**。
- 后续所有行为（调度、司机执行、Trip、财务）都以这张运单为中心展开。

---

### 2. Excel 业务数据与真实业务起点

- **两个关键 Excel 文件**：
  - `2026年转运汇总表.xlsx`：记录全年实际发生的转运/派送业务明细。
    - 维度：柜号、所在仓库、进仓方式、到仓日、卸柜情况、分货 SKU、件/箱数、方数、目的仓（YYZx/YOWx 等）、派送指令等。
    - 本质：**事实数据（实际发生过的每一票/每一托运输）**。
  - `合作单位报价卡.xlsx`：记录各合作车队/仓的报价规则。
    - 不同 Sheet 对应不同合作方（JW、ESC、无忧达、Andy…）。
    - 维度：仓号、板数区间（1–4板、5–14板、15–28板、散板）、目的仓、运输类型（直卡 / 拖车）、超时费、贴标费、私人地址加价等。
    - 本质：**价格配置 / 合同价表**。

- **真实业务起点（没有系统时代）**：
  - 货到仓 → 运营同事在 Excel（转运汇总表）里新增一行记录：
    - 标出柜号、所在仓、到仓时间、目的仓、件数/托数/方数等；
  - 需要转运时，再参考报价卡，人工计算每一票或每一托转运的报价与成本。

**结论（从 Excel 出发）：**

- 业务现实的起点是：**“某票货/托盘需要从仓库 A 转到目的仓 B，被运营同事记入转运 Excel 的那一行”**。
- 现有 Excel 主要是“系统出现前的业务流水记录”，而不是当前系统内的真实数据源。

### 3. 现有系统能力与缺口（与测试文档/Excel 对照）

1. **系统已实现的部分**
   - Waybill（运单）、Trip（行程）、Driver/Vehicle（司机/车辆）、Financials（财务）等实体已存在。
   - 后端 API 已覆盖：
     - 创建运单、指派司机车辆、Trip 跟踪；
     - 状态更新、POD/签收、消息/事件上报；
     - 财务应收/应付记录生成。
   - 测试计划中的 P0 闭环在代码层面是可以跑通的。

2. **系统缺失的关键环节**
   - **没有“转运单”这一层业务抽象**：
     - 当前“运单详情”只体现一票运输任务的汇总信息；
     - 没有与 Excel 一致的“多行货物/托盘明细”结构和界面。
   - **没有运营录入入口**：
     - 运营同事现在不能在系统中一票一票录入“转运单”；
     - 历史 Excel 数据只能间接导入为某种汇总记录，无法一行一行对齐管理。
   - **缺少托盘级 HOLD 管理能力**：
     - Excel 中存在“暂不处理 / RMB 结算暂不处理 / HOLD”之类备注；
     - 系统目前没有托盘级状态（HOLD / RELEASED）及其仓库/原因记录。
   - **报价卡未结构化接入**：
     - 报价规则仍停留在 Excel；
     - Waybill 的自动报价与实际合作单位报价卡之间尚未打通。

---

### 4. 当前对“业务起点”的统一理解

综合测试文档 + Excel + 系统现状，形成以下统一认识：

- **目标形态（TMS 最终希望达到的模型）**：
  - 业务起点：**运营同事在系统里创建一张“转运单”**：
    - 录入头信息（柜号、所在仓、到仓日、合作方…）；
    - 在转运单下添加多行货物/托盘明细；
    - 明细上可以标记 HOLD，绑定报价规则，选择何时/如何生成运单。
  - 从这张“转运单”出发，系统自动：
    - 对接报价卡算出建议报价与成本；
    - 按目的仓/合作方/时间生成一张或多张运单；
    - 再进入调度、Trip、司机执行与财务结算链路。

- **当前版本的折中现实**：
  - 测试与代码层面，默认业务起点 = **已有运单（Waybill 已存在）**；
  - “转运单 + 多行明细 + HOLD 管理 + 报价卡联动”这一层仍是缺失的。

---

### 5. 为后续测试设计提供的锚点

基于以上理解，后续在补齐功能 & 设计测试时需要明确：

1. **功能侧锚点**
   - 新功能需要显式引入：
     - 转运单（Transfer Order）头 + 多行明细模型；
     - 托盘/明细级 HOLD 管理；
     - 对接报价卡的在线算价；
     - 从“转运单明细”批量/按组生成运单。

2. **测试侧锚点**
   - 业务起点测试场景 = “一票转运需求在系统内被创建”的那一刻：
     - 既可通过“运营手填转运单”来触发；
     - 也可通过测试专用的 seed/脚本生成等方式来模拟。
   - 完整闭环 = **转运单 → 运单 → Trip → 签收 → 财务**：
     - 需要在现有 P0 Flow 的基础上，把“转运单录入 + 明细/HOLD 管理 + 报价卡匹配”这些前置步骤补入。
