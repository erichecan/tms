### 二、后端修改记录（核心功能）

1. **转运单与明细后端（v0.1 概述）**
   - 新增 Transfer Order 相关表与 API（简要）：
     - 表：`transfer_orders`、`transfer_order_lines`（字段按 PRD：partner、container_no、warehouse、entry_method、arrival_date、currency、hold_status...）。
     - API 路由（`/transfer-orders` 下）：
       - `GET /transfer-orders`：列表（状态、柜号、仓库、合作单位等筛选）。
       - `GET /transfer-orders/:id`：详情（头 + 明细）。
       - `POST /transfer-orders`：创建。
       - `PUT /transfer-orders/:id`：更新头信息。
       - `DELETE /transfer-orders/:id`：逻辑删除/作废。
       - `POST /transfer-orders/:id/lines`：批量 upsert 明细（create + update）。
       - `POST /transfer-orders/:id/generate-waybills`：从选中明细生成运单。
     - Waybill 记录中增加来源信息：
       - 在 `details` JSON 或关联表中保存 `transfer_sources`（包含 transfer_order_id、line_id、pallets、cbm）。
   - 权限：
     - 所有 `/transfer-orders` 相关路由均挂在 `verifyToken` 下，限制为 dispatcher/admin 角色使用。

2. **批量生成运单逻辑改造（基于集装箱 & FC 目的地）**
   - 文件：`apps/backend/src/controllers/ContainerController.ts`
   - 功能点：
     1. **标准化起点/终点地址**
        - 起点 `origin`：
          - 优先使用 `containers.warehouse_id` 在 `fc_destinations` 中查找匹配 code，若存在则拼接 `address + city + province` 作为完整地址；
          - 否则退回到原始 `warehouse_id` 字符串。
        - 终点 `destination`：
          - 使用 `container_items.dest_warehouse` 在 `fc_destinations` 中查找 code；
          - 若存在则同样用 `address + city + province` 组合，否则退回 `delivery_address` 或 `dest_warehouse`。
     2. **对接 `pricing_matrices` 计算 Revenue**
        - 仍按 `customer_id + destination_code + vehicle_type + pallet_tier` 查询 `pricing_matrices`；
        - 找到则填充：
          - `price_estimated = base_price (+ per_pallet_price * 超额托数)`；
          - `pricing_matrix_id` 存入运单；
          - 额外标记 `hasPricing = true`。
        - 未找到时：
          - `price_estimated` 保持为 0；
          - 在返回给前端的 `generatedWaybills[]` 元素中标记 `pricingMissing: true` 以便 UI 给出提示。
     3. **插入运单记录字段**
        - 关键字段：
          - `origin`：上述算出的仓库完整地址；
          - `destination`：目的 FC 完整地址或备用文本；
          - `cargo_desc`：`FBA: {fba_shipment_id or sku} | {piece_count}pcs`；
          - `pallet_count`：根据 `pallet_count_num` 计算；
          - `price_estimated`：来自 `pricing_matrices`，否则为 0；
          - `fulfillment_center`：`dest_warehouse`。

3. **FC 目的地与费率矩阵 CRUD（已有能力确认）**
   - 控制器：`apps/backend/src/controllers/PricingMatrixController.ts`
     - `pricing_matrices`：
       - `GET /api/pricing/matrices`：按客户聚合概览。
       - `GET /api/pricing/matrices/:customerId`：单客户所有费率（可含归档）。
       - `POST /api/pricing/matrices`：单条 upsert（存在则更新，不存在则插入）。
       - `POST /api/pricing/matrices/batch`：批量 upsert。
       - `DELETE /api/pricing/matrices/:id`：标记为 `ARCHIVED`。
     - `fc_destinations`：
       - `GET /api/pricing/fc-destinations`：获取全部 FC 字典。
       - `POST /api/pricing/fc-destinations`：按 code upsert（不存在创建，存在更新）。
   - 迁移脚本：`apps/backend/src/migrate.ts`
     - 已创建 `fc_destinations` 与 `pricing_matrices` 表结构，并填充示例 FC 代码。

### 三、前端修改记录（核心功能）

1. **转运单 UI：创建/编辑/删除/明细管理**
   - 文件：`apps/frontend/src/TransferOrderDetail.tsx`
   - 能力：
     - 头部表单：合作单位、柜号/提单号、所在仓库、到仓日期、主目的仓、币种、整体备注等。
     - 明细表：
       - 行内编辑字段：SKU、FBA、件数、托数、方数、目的仓、合作单位、HOLD 状态与原因。
       - 行级操作：
         - 标记/解除 HOLD（按钮 + `hold_status` 下拉）。
         - 拆分明细（复制一行，分配托数与剩余托数/方数）。
         - 删除明细行（软删除，保留在前端状态，保存草稿时同步至后端）。
       - 批量勾选可生成运单的明细（仅 `HOLD != PENDING` 且剩余托数 > 0 的行可选）。
     - 顶部按钮：
       - **保存草稿**：新建走 `createTransferOrder` + `saveTransferOrderLines`，编辑走 `updateTransferOrder` + `saveTransferOrderLines`。
       - **删除/放弃新建**：
         - 新建未保存时：`window.confirm` 后直接返回列表。
         - 已存在转运单时：`window.confirm` 后调用 `DELETE /transfer-orders/:id`，删除成功后返回列表。
     - 生成运单按钮：
       - 调用 `POST /transfer-orders/:id/generate-waybills`，仅传入勾选行的 `line_ids`。

2. **集装箱转运管理 UI：移除独立“排预约” + 提示缺少报价**
   - 文件：`apps/frontend/src/ContainerManagement.tsx`
   - 改动：

       - 删除 `showAppt` 状态与预约 Modal；
       - 删除每条 `container_item` 上的「📅 排预约」按钮；
       - 预约逻辑后续由“指派司机/车辆 + Trip 流程”来驱动，不再二次手填“司机代码”。
     - **改善总方数统计健壮性**：
       - 将 `(selected.total_cbm || 0).toFixed(2)` 改为 `Number(selected.total_cbm || 0).toFixed(2)`，兼容字符串/undefined。

### 四、测试建议（本地 + GCP）

#### 4.1 本地验证建议（优先做完）

1. **转运单基础功能**
   - 登录本地前端（`http://localhost:5173`，dispatcher 账号）。
   - 进入「转运管理 → 转运单」：
     - 新建转运单：
       - 填写头部字段（合作单位、柜号、仓库、到仓日期等）。
       - 添加若干明细行（SKU/FBA/托数/方数/目的仓/HOLD）。
       - 测试行级操作：HOLD、拆分、删除。
     - 点击“保存草稿”：
       - 确认转运单出现在列表中；
       - 重新打开详情，明细保持一致。
     - 使用“删除转运单”按钮：
       - 新建未保存：确认后返回列表，列表中不应出现该单；
       - 已存在转运单：确认后列表中不再显示该单，后端状态为作废（按实现）。

2. **集装箱 → 批量生成运单**
   - 在「转运管理（集装箱视图）」：
     - 选择一个已有集装箱，展开右侧详情；
     - 点击“批量生成运单”：
       - 若 `pricing_matrices` 中已有对应客户+目的仓组合：
         - 在运单列表中查看新生成 Waybill，`price_estimated` 应为非 0，`origin/destination` 为完整地址；
       - 若缺少配置：
         - 前端弹窗应明确列出缺少配置的目的仓 code；
         - 运单列表中对应记录 Revenue = 0。

3. **报价管理联动**
   - 在「报价管理 → FC 目的地」：
     - 新增或编辑一个 FC 代码的地址信息；
     - 再次批量生成运单，检查：
       - 该目的仓的运单 `destination` 是否更新为新配置的地址；
   - 在「报价管理 → 费率矩阵」：
     - 为特定客户 + FC 目的仓配置 1–4/5–13/14–28 档位价格；
     - 再次批量生成运单，确认：
       - 相应运单的 `price_estimated` 变为非 0；
       - 缺少配置的提示不再包含该目的仓。
